## 1. Relational Database Design Principles
- **Declarative Immutability (`ON DELETE RESTRICT`):** To preserve clinical audit non-repudiation under **NEN 7510 / NEN 7512** and Dutch regulatory IGJ mandates, legacy `ON DELETE CASCADE` or `SET NULL` operations have been entirely expunged. Foreign keys linking core clinical assets (`fall_events`, `medication_reminders`, `device_health_events`, `carer_handover_notes`) reference user entities with strict `RESTRICT` operations.
- **Atomic GDPR Profile Redaction (`soft_purge_profile`):** When an older adult requests Right to Erasure (GDPR Art. 17), active transactional links cannot be deleted due to `RESTRICT` immutability guards. Instead, `soft_purge_profile(p_target_id)` atomically re-anchors historical `webhook_receipts`, `medication_ocr_reviews`, and `carer_handover_notes` to the **Anonymous System Sentinel** node (`00000000-0000-0000-0000-000000000001`), executes exact free-text plain-text clinical narrative scrubbing (`content_nl := '[ERASED] Documentatie overgedragen of verwijderd op verzoek van betrokkene per AVG Art. 17'`), overwrites explicit direct PII (`status := 'erased'`, `full_name := 'Geanonimiseerd'`), and entirely nullifies 1536-dimensional `pgvector` mathematical OpenAI embeddings in `companion_memory`.
- **Zero-WAL PSD2 Open Banking Buffering:** Direct relational `INSERT` operations on financial webhooks collided with GDPR soft-deletions and induced transactional deadlocks (*40001*). The data layer mitigates this by maintaining a stable secondary key ledger (`profiles_snapshot`) that is never mutated during GDPR cleanups, while real-time PSD2 webhooks write to an `UNLOGGED` scratch table (`psd2_webhook_ingress_buffer`) with exactly `0%` WAL disk overhead.
- **Instantaneous PostGIS Range Partitioning (`location_events_partitioned`):** PostGIS emergency spatial indices (`GIST`) are range-partitioned exactly on `created_at` (`PARTITION BY RANGE (created_at)`). Future sliding days of child partitions are pre-created, allowing historical telematics cleanups to execute entirely via zero-load child partition drops (`DROP TABLE ... CASCADE` -> `0.00ms` lock duration).
